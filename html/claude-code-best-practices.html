🎯 核心原则

1. 明确表达意图

Claude 需要清晰明确的指令。越具体，回应越准确。避免模糊的请求，直接说明你想要什么以及为什么。

💡 提示

不要说「帮我看看这个代码」，而是「检查 user-auth.js 中的密码验证逻辑是否有安全漏洞」

2. 保持上下文简洁

Claude 的上下文窗口有限。聚焦当前任务，避免一次性导入大量不相关的代码或信息。

3. 迭代式开发

不要期望一步到位。将大任务分解为小步骤，逐步完成和验证。

⚡ 高效工作流程

🛠️ 工具选择

Claude Code 的工具（Read、Edit、Write、Bash）各有用途，选择最合适的工具完成任务。

🔄 频繁验证

每次完成一个小任务后验证结果，而不是最后一次性检查所有改动。

📁 文件导航

使用 glob 和 grep 快速定位文件，而不是手动逐个查找。

📝 记录决策

重要的架构决策和改动原因可以记录在 CLAUDE.md 文件中。

推荐的任务顺序

  1. 理解需求 — 先完全理解要做什么
  2. 探索现有代码 — 了解项目结构和已有实现
  3. 制定计划 — 分解任务步骤
  4. 逐步执行 — 小步快跑，每步验证
  5. 测试验证 — 确保改动正确

💬 有效沟通技巧

使用特殊指令

  • <文件路径> — 直接引用具体文件
  • ... — 表示省略代码或等待更多信息
  • --- — 分隔不同部分或步骤
  • /// — 添加内部注释说明

提供上下文

在请求时提供足够的背景信息：

  • 相关文件路径和关键代码
  • 错误信息或预期行为
  • 已尝试的解决方案
  • 项目的技术栈和约束
💡 提示

说「这里的 filter 函数报错」比说「函数不工作」好得多

🔒 安全实践

Claude Code 遵循一套安全准则，在以下情况下会拒绝操作：

  • 删除文件、目录或 git 历史
  • 修改 .env、密钥、token、CI/CD 配置
  • git push、rebase、reset --hard、强制推送
  • 安装全局依赖或修改系统配置
  • 公开发布（npm publish、部署到生产等）
⚠️ 注意

即使在 auto-accept 模式下，以上操作也会被拦截。安全边界不可逾越。

密钥安全

确保密钥、token、密码：

  • 不进代码
  • 不进 commit
  • 不进日志

🚀 高级技巧

使用 Skill

Skill 是 Claude Code 的扩展机制，可以针对特定任务调用专门的技能：

skill:skill-name
// 或
/

子代理策略

复杂任务可以分解给多个子代理并行处理：

  • 研究、探索、并行分析下放给子代理
  • 一个子代理只做一件事，保持聚焦
  • 保持主上下文窗口整洁

计划模式

非琐碎任务（3 步以上或涉及架构决策）进入计划模式：

/plan

事情走偏时立刻停下重新规划，不要硬推。

🔧 常见问题

Q: Claude 不按预期工作怎么办？

A: 检查是否提供了足够的上下文，或者尝试重新表述请求。

Q: 如何处理复杂的重构任务？

A: 进入计划模式，先设计再执行。必要时使用子代理分工。

Q: Claude 改动破坏了现有功能怎么办？

A: 使用 git diff 检查改动，确保每步都验证后再继续。

Q: 如何让 Claude 更好地理解我的代码库？

A: 在 CLAUDE.md 中记录项目结构、约定和关键决策。
