Cursor 设计师 Ryo Lu：软件曾有过灵魂

Ryo Lu 发出了一篇深度反思：2005 年左右的 Mac 体验——Dock 弹跳、Genie 效果、Exposé 视窗管理——让人觉得在触碰某种活着的东西。那时候的软件有质感、有个性、有哲学。

后来发生了什么？A/B 测试磨平了棱角，设计系统标准化了性格，优化让一切更快、更顺、更一致——却也更无趣。每个 App 看起来都一样，每个产品都用同一种冷静、无摩擦的声音说话。

然后 AI 来了，速度变成了非人的速度。一个下午生成一个产品，午饭前上线一个功能。想法到代码的gap基本为零了。

但问题也来了：做得太容易，平庸之作也免费涌入了。它们看起来没问题，能跑，能通过 review，只是现在无限多。互联网正在被那些功能正确但毫无意义的软件填满。

Ryo Lu 的核心观点是：AI 不会取代真正在意的人，但它会把他们淹没。品味不是 markdown 技巧，关心不是参数。你放进一个东西里的那种奇怪的、具体的”灵魂”，无法被程序化。

真正正确的方向，不是让垃圾产出得更快，而是终于给有真正 vision 的人工具，去做他们一直想象却无法独自构建的东西。如果做对了，我们得到的不是更快的工厂，而是一场复兴。

Vercel CEO Guillermo Rauch：Opus 4.5 是通向新工程方式的一扇单向门

Rauch 分享了 Vercel 内部的经验：Opus 4.5 发布后，Agent 现在已经承担了他们大部分的编码工作。

但他们也给团队发出了明确信号：Vibing 和关键基础设施不能共存。他知道 LLM 固有的缺陷和过度自信，所以 Vercel 正在分享他们早期关于”负责任地 Agent 化”（agenting responsibly）的内部指导原则，优先考虑安全性、耐久性和可用性。

这和 Ryo Lu 的担忧形成了有趣的呼应：AI 让编码变得容易，但容易和可靠是两回事。

Box CEO Aaron Levie：企业 AI Agent 落地没有捷径

Levie 写了一段很扎实的分析，揭示了为什么 AI Agent 在企业落地的难度被低估了。

他说，很多人看到编程 AI 的收益就想当然认为知识工作 AI 也会同样轻松——但这是两个完全不同的挑战。

编程 AI 为什么容易成功：Agent 获取上下文容易，用户是技术型的，代码结构清晰，输出可验证。

知识工作 AI 落地为什么难：数据分散，权限混乱，信息没有 AI-ready 的结构。需要有人去：

  • 把非结构化数据整理成 Agent 能访问的格式
  • 学习工作流程和流程，创建 Agent 可用的 skills 或 plans
  • 把不同系统连接起来
  • 可能还要改变流程本身以适应 Agent
  • 然后设计人在哪里介入、如何验证

结论：这些工作没有捷径，必须由团队里的人来完成。但这也意味着企业内会出现大量新角色，专攻这个领域的人将在经济中获得巨大价值。对于早期职业者来说，这是一个快速产生重大影响的好机会。

Andrej Karpathy：npm axios 供应链攻击的亲身经历

Karpathy 报告了一个新的供应链攻击——这次针对 npm axios（最流行的 HTTP 客户端库，每周下载量 3 亿次）。

他在系统中扫描发现，几天前实验 gmail/gcal cli 时导入了一个来自 googleworkspace/cli 的依赖。已安装的版本（幸运地）解析到了未受影响的 1.13.5，但项目依赖没有锁定版本——如果他当天早些时候做这件事，代码会解析到 latest，然后就会被黑。

他的判断：pip、npm 等包管理工具的默认设置必须改变，单个感染不应该通过未锁定版本传播到用户。个人防御可以部分防御（本地设置 release-age 约束，或使用容器），但最终必须从源头解决问题。
